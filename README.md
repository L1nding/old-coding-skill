# 古法编程

`old-coding-skill` 是一个面向 Codex 的工程协作技能，用于约束 AI 参与软件开发的方式，避免开发流程滑向 AI 黑盒协作与 vibe coding。

本技能强调一种明确的人机分工：

- 人类负责问题定义、方案取舍、架构决策、边界设定与最终验收
- AI 负责资料整理、方案比较、白盒解释、风险审计、实现辅助与验证辅助

核心目标不是拒绝 AI，而是防止 AI 取代工程判断，确保开发过程始终可解释、可验证、可追责。

## 核心理念

古法编程强调以下原则：

- 先建模，再实现
- 先设计，再编码
- 先解释，再采纳
- 先验证，再放心

在这一模式下，任何准备进入代码库的内容都应当能够回答以下问题：

1. 为什么这样设计
2. 运行时如何执行
3. 风险和边界在哪里
4. 如何验证其正确性

## 解决的问题

本技能主要针对以下失控协作现象：

- 根据一句模糊需求直接生成大段代码
- 代码可以运行，但设计原因不清楚
- AI 替代人类做架构决策
- 缺少方案比较、边界说明和验证闭环
- 开发者逐渐退化为复制、粘贴、运行的执行角色

## 工作流

古法编程模式默认按照以下顺序推进任务：

1. 问题建模
2. 方案比较
3. 设计决策
4. 受控实现
5. 白盒审计
6. 验证闭环

这一流程要求在实现前先明确问题本质、约束条件、候选方案和设计边界，在实现后补齐白盒解释、风险审计与验证结论。

## 模式说明

### 原理反问模式

默认模式。用于在解释或实现完成后继续追问设计原因、执行路径、边界条件和验证方式，防止代码“经过屏幕，却没经过大脑”。

### 白盒审计模式

用于审查已有代码、设计或方案，重点分析执行路径、状态变化、副作用、风险点以及复杂度成本，而不是给出泛泛而谈的“看起来不错”式评价。

### 代码临摹模式

用于教学和训练场景，通过逐行展示、逐行解释、逐行模仿的方式帮助使用者真正理解代码，而不是直接接受完整结果。

## 文件说明

- `SKILL.md`
  技能主定义，规定古法编程的定位、工作流、禁止行为和输出结构。
- `references/anti-black-box-workflow.md`
  说明如何把 AI 从黑盒产码机约束回受控的工程辅助角色。
- `references/modes-and-prompts.md`
  说明原理反问、白盒审计、代码临摹三种模式的使用方式与推荐提示语。
- `agents/openai.yaml`
  提供展示名、简述与默认提示语等界面元数据。

## 适用场景

本技能适合用于以下类型的任务：

- 需要先做问题建模和方案比较的开发任务
- 希望由人类掌控设计、由 AI 提供辅助的协作场景
- 需要白盒解释、风险分析或原理审计的代码任务
- 希望避免“先生成再理解”的开发方式

## 输出结构

启用本技能时，推荐采用以下输出顺序：

1. 问题本质
2. 候选方案
3. 设计决策
4. 实现建议
5. 白盒解释
6. 风险与边界
7. 验证方式
8. 原理反问

## 设计来源

本技能的思路启发自仓库 [zjw-swun/old-code](https://github.com/zjw-swun/old-code) 中关于“原理反问”“代码临摹”“反 AI 黑盒”的核心表达。在此基础上，当前版本针对 Codex 的技能形态进行了重写，更强调工程流程约束、人机职责分离以及验证闭环。
