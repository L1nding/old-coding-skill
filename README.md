# 古法编程

`old-coding-skill` 是一个用于反对 AI 黑盒协作和 vibe coding 的 Codex 技能。

它保留“古法编程”这个名字，但核心不是怀旧，也不是拒绝 AI，而是把人类重新放回软件开发中的主导位置：

- 人类负责设计
- 人类负责取舍
- 人类负责验收
- AI 只负责辅助

## 它解决什么问题

这个技能主要解决下面这些失控协作问题：

- 一句需求直接生成一大段代码
- 代码能跑，但人说不清为什么这么写
- AI 直接替人做架构决策
- 没有方案比较、没有边界说明、没有验证闭环
- 人类逐渐退化成复制、粘贴、运行的执行器

如果你想要的是“人主导设计，AI 辅助实现”，这个技能就是为这个目的写的。

## 核心主张

- 不反对 AI，反对把 AI 当黑盒替身
- 不反对生成代码，反对跳过建模、设计和验证
- 不接受“看起来没问题”，必须追问“为什么这样设计”
- 不让人类做代码接收者，而要做人机协作中的设计者和裁决者

## 默认工作流

这个技能默认要求按以下顺序推进：

1. 问题建模
2. 方案比较
3. 设计决策
4. 受控实现
5. 白盒审计
6. 验证闭环

也就是说，先想清楚，再写；先解释，再采纳；先验证，再放心。

## 三种模式

### 1. 原理反问模式

默认模式。实现或解释完成后，不直接收尾，而是继续追问：

- 为什么这么设计
- 运行时怎么执行
- 风险和边界在哪里
- 如何证明它是对的

### 2. 白盒审计模式

适合 review、解释、分析已有代码或方案。重点不是“夸代码”，而是拆开看：

- 执行路径
- 状态变化
- 副作用
- 风险点

### 3. 代码临摹模式

适合教学和训练。不是直接给完整答案，而是逐行解释、逐行理解、逐行模仿，让代码真正经过大脑。

## 文件说明

- `SKILL.md`
  技能主定义，规定古法编程的立场、工作流、禁止行为和输出结构。
- `references/anti-black-box-workflow.md`
  说明如何把 AI 从黑盒产码机拉回受约束的工程辅助角色。
- `references/modes-and-prompts.md`
  说明原理反问、白盒审计、代码临摹三种模式的使用方式和提示语。
- `agents/openai.yaml`
  提供展示名、短描述和默认提示词等界面元数据。

## 适用请求示例

- “别直接生成代码，先做设计。”
- “我要反 vibe coding。”
- “你辅助我，不要替我做架构决策。”
- “请白盒解释这段代码。”
- “实现后继续反问我原理。”
- “让我做人类设计者，AI 只做辅助。”

## 使用方式

把这个目录放到 Codex 的 skills 目录后，在需要强约束 AI 协作方式的任务中触发它。

推荐输出顺序是：

1. 问题本质
2. 候选方案
3. 设计决策
4. 实现建议
5. 白盒解释
6. 风险与边界
7. 验证方式
8. 原理反问
