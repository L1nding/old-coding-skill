# 古法编程技能

`old-coding-skill` 是一个面向遗留系统、历史项目和陌生代码库的 Codex 技能。它的目标不是“直接改代码”，而是先帮助你看清系统当前状态、识别高风险改动面，再以更稳妥的方式完成修改。

## 这个技能解决什么问题

它主要用于下面几类场景：

- 接手老项目，但缺少完整文档
- 需要先梳理当前架构，再决定从哪里下手
- 想定位一段行为到底由谁负责、会影响谁
- 想在不破坏隐藏逻辑的前提下做小步修改
- 想把“我觉得是这样”变成“我有证据证明现在就是这样”

## 核心思路

这个技能的默认流程是：

1. 先建立系统快照
2. 再识别安全改动面
3. 然后按白盒方式实施修改
4. 最后用由窄到宽的验证方式确认结果

它强调几件事：

- 优先相信代码、日志、测试和运行产物这些一手证据
- 明确区分“已验证”和“推断”
- 默认做最小改动，而不是一上来就大重构
- 修改前先讲清楚执行路径、状态归属和副作用

## 目录结构

```text
old-coding-skill/

│   └── openai.yaml
└── references/
    ├── system-snapshot.md
    └── change-playbook.md
```

## 文件说明

- `SKILL.md`
  技能主定义，包含触发说明、工作流、输出结构和注意事项。
- `references/system-snapshot.md`
  用来快速梳理系统当前状态的检查清单。
- `references/change-playbook.md`
  用来指导如何安全修改遗留系统的操作手册。
- `agents/openai.yaml`
  提供展示名、短描述和默认提示词等界面元数据。

## 适用请求示例

- “看懂这个系统”
- “帮我梳理当前状态”
- “接手这个老项目，先别改代码”
- “分析这条运行链路”
- “先定位改动点，再动手”
- “这个历史系统我想安全地改一下”

## 使用方式

把这个目录放到 Codex 的 skills 目录后，就可以在相关请求中直接触发它。

默认推荐让它先输出：

1. 当前状态
2. 风险地图
3. 改动方案
4. 验证结果

这样比“直接让我改”更适合接手复杂系统，也更能避免误改。

## 设计取向

这个技能借鉴了“古法编程”的思想，但更偏向工程实践，而不是口号式表达。它不要求排斥 AI，而是要求在使用 AI 时保留对白盒执行路径、系统边界和验证证据的掌控。
